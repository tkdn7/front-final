# 08 반응형 테스트 수정

## 1. 사용자 프롬프트 원문
LIFE CYCLE 앱의 주요 기능을 구현한 뒤, 사이트맵과 비교하면서 화면과 기능을 점검해줘. 특히 과제 보고서에 캡처를 넣어야 하므로 데스크톱과 모바일에서 화면이 깨지지 않는지 확인하고 수정 계획을 세워줘.

점검 기준은 다음과 같다.
- 데스크톱에서 메인 대시보드가 사이트맵처럼 TODO/습관, 캘린더, AI 브리핑 영역으로 보이는지 확인한다.
- 모바일에서 nav와 각 패널이 세로로 자연스럽게 배치되는지 확인한다.
- 긴 TODO 제목, 긴 일정 제목, 긴 일기, 긴 AI 답변이 영역 밖으로 넘치지 않는지 확인한다.
- TODO와 습관 목록이 많아질 때 내부 스크롤이 되는지 확인한다.
- 캘린더 월 이동 버튼이 정상 동작하는지 확인한다.
- 날짜 클릭 후 일정과 한 줄 일기 영역이 나타나는지 확인한다.
- 라이트/다크 모드 토글이 정상 동작하는지 확인한다.
- API 키 입력 없이 AI 브리핑을 실행했을 때 안내 메시지가 나오는지 확인한다.
- 잘못된 API 키, 잘못된 지역명, 잘못된 모델명에서 서로 다른 오류 메시지가 나오는지 확인한다.
- API 요청 후 키가 localStorage, sessionStorage, URL, 화면에 남지 않는지 확인한다.

출력은 발견한 문제, 수정 방법, 보고서에 넣을 캡처 후보, 최종 확인 체크리스트로 나눠줘.

## 2. AI 답변 요약
- 데스크톱과 모바일 화면을 각각 확인해 레이아웃 깨짐을 찾는다.
- 사이트맵과 다르게 보이는 부분을 우선 수정 대상으로 둔다.
- 긴 텍스트는 줄바꿈, 스크롤, 최대 높이 설정으로 처리한다.
- TODO/습관/AI 답변 영역은 내부 스크롤을 적용한다.
- API 오류는 키 누락, 키 오류, 지역명 오류, 모델 오류로 나누어 보여준다.
- API 키가 저장소나 URL에 남지 않는지 확인한다.
- 보고서용 캡처는 메인, 캘린더 선택 상태, API 키 입력 모달, AI 브리핑 결과, 오류 상태를 추천한다.

## 3. AI 활용이 필요한 문제 상황
기능이 동작하더라도 화면이 깨지거나 보고서 캡처에 부적절한 상태가 나오면 제출물 품질이 낮아질 수 있었다. 여러 화면 크기와 오류 상황을 한 번에 점검하기 위해 AI에게 체크리스트와 수정 기준을 요청했다.

## 4. 채택한 내용
- 데스크톱과 모바일을 모두 점검한다.
- 사이트맵과 다른 부분을 우선 수정한다.
- 긴 텍스트와 스크롤 문제를 따로 확인한다.
- API 성공 화면뿐 아니라 오류 화면도 보고서 자료로 남긴다.
- API 키가 남지 않는지 저장소와 화면을 점검한다.

## 5. 거부하거나 수정한 내용
- AI가 데스크톱 화면만 확인하자고 하면 수정한다. 반응형은 과제 완성도에 중요하다.
- AI가 오류 화면을 생략하고 성공 화면만 보고서에 넣자고 하면 수정한다. 오류 처리도 구현 근거가 된다.
- AI가 긴 텍스트를 숨김 처리만 하자고 하면 수정한다. 사용자가 내용을 읽을 수 있게 줄바꿈 또는 스크롤을 적용한다.
- AI가 사이트맵과 다르게 보이는 부분을 디자인 취향으로 넘어가면 수정한다. 사이트맵 기준에 맞춰야 한다.
- AI가 실제 API 키가 보이는 캡처를 사용하자고 하면 거부한다. 보고서 캡처에서는 키를 비우거나 마스킹한다.

## 6. 내가 직접 해결한 부분과 배운 점
테스트는 단순히 버튼을 눌러보는 것이 아니라, 사용자가 실제로 실패하거나 긴 데이터를 입력했을 때도 화면이 유지되는지 확인하는 과정이라는 점을 배웠다. 특히 보고서에는 성공한 결과뿐 아니라 어떤 기준으로 수정했는지도 남길 수 있어야 한다고 판단했다.

실제 Chrome 렌더링 확인 과정에서 모바일 nav의 `습관`, `일기` 글자가 두 줄로 쪼개지고 AI 버튼이 오른쪽에서 일부 잘려 보이는 문제를 발견했다. 이 부분은 AI가 자동으로 완성했다고 보기보다, 캡처 품질과 사용성을 기준으로 직접 판단해 CSS를 수정한 사례로 기록한다. 모바일에서는 nav를 5칸 그리드로 바꾸고 버튼 최소 너비를 줄여 모든 메뉴가 한 줄 안에 보이도록 직접 조정했다.

## 7. 다음 단계
지금까지의 프롬프트 기록, 채택/거부 기준, 구현 결과를 과제 조건의 AI 활용 보고서 문장으로 정리한다.

## 8. 실제 진행 시나리오
이 단계에서 최종 화면 캡처 후보를 정하고, 수정 전후 판단 근거를 남긴다. 실제 검증은 로컬 `index.html`을 Chrome으로 열어 데스크톱 1440px, 모바일 390px 화면을 확인하는 방식으로 진행했다. 검증 중 모바일 nav 잘림 문제를 발견해 `styles.css`의 모바일 미디어쿼리를 수정했고, 재검증 결과 nav 버튼 overflow가 0개, 콘솔 오류가 0개로 확인되었다. 다음 단계에서는 이 기록들을 묶어 AI 활용 보고서 PDF에 들어갈 문장으로 바꾼다.

## 9. 후속 수정 반영
후속 요구사항에서 모바일 화면은 테스트하거나 열지 않기로 결정했기 때문에, 최종 구현에서는 모바일 전용 미디어쿼리를 제거하고 데스크톱 화면 기준으로 레이아웃을 고정했다. 따라서 이전 모바일 nav 수정은 중간 과정으로 기록하고, 최종 보고서에서는 “초기에는 모바일도 검토했지만 제출 기준을 데스크톱으로 좁히면서 데스크톱 밀도를 높이도록 직접 수정했다”고 설명한다.

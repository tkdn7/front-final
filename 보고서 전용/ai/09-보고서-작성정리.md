# 09 보고서 작성 정리

## 1. 사용자 프롬프트 원문
지금까지 LIFE CYCLE 웹앱을 만들기 위해 AI에게 요구사항 분석, 사이트맵 해석, 프로젝트 구조 설계, 메인 대시보드 구현, localStorage 기능, API 키 입력 방식, 실제 API 연동, 반응형 테스트를 단계별로 요청했다.

이제 과제 조건의 `AI 활용 문서화(필수)` 항목에 맞게 AI 활용 보고서에 들어갈 내용을 정리해줘.

반드시 포함할 항목은 다음과 같다.
1. 어떤 부분에서 AI를 활용했는지
   - AI 활용이 필요한 문제 상황
   - 실제 사용한 프롬프트 내용
2. AI가 제안한 솔루션 중 채택/거부한 것과 그 이유
   - 프롬프트에 대한 AI 답변 요약
   - 답변 분석을 통해 실제로 활용한 내용
   - 거부하거나 수정한 내용과 이유
3. AI 활용 과정에서 배운 점
   - 스스로 해결한 문제
   - 직접 판단한 기준
   - 프로젝트를 진행하며 배운 점

특히 다음 사례를 중요하게 정리해줘.
- 사이트맵을 최우선 구현 기준으로 삼은 점
- 구상 초안본의 아이디어 중 사이트맵에 없는 기능은 제외하거나 후순위로 둔 점
- React/Tailwind 구조도 가능했지만 최종 구현은 HTML/CSS/JS 정적 구조로 직접 조정한 점
- API 키를 `.env`나 localStorage에 저장하지 않고 사용자가 매번 입력하도록 수정한 점
- AI 답변을 그대로 사용하지 않고 과제 조건, 보안, 사이트맵 기준에 맞게 선별한 점
- 오류 화면과 반응형 테스트까지 보고서 자료로 남기기로 한 점

문장은 보고서에 바로 붙여 넣을 수 있게 작성해줘. 단, 실제 API 키나 민감 정보는 포함하지 말아줘.

## 2. AI 답변 요약
- AI 활용 보고서는 과제 조건의 세 항목을 그대로 목차로 사용한다.
- 첫 번째 항목에는 요구사항 분석, 사이트맵 해석, 컴포넌트 설계, API 흐름 설계, 테스트 체크리스트에 AI를 활용했다고 정리한다.
- 두 번째 항목에는 채택한 제안과 거부/수정한 제안을 표 형태로 정리한다.
- 채택 사례는 사이트맵 우선 구현, 컴포넌트 분리, localStorage 저장, API 키 매번 입력 방식, 오류 상태 구분이다.
- 거부/수정 사례는 API 키 저장, 과도한 기능 추가, 사이트맵에 없는 화면 추가, React/Tailwind 고정 전제, 성공 화면만 캡처하는 방식이다.
- 세 번째 항목에는 AI가 제안해도 최종 판단은 사용자가 해야 한다는 점을 배운 내용으로 쓴다.
- 보고서에는 프롬프트 원문 전체와 답변 요약을 단계별 부록처럼 연결할 수 있다.

## 3. AI 활용이 필요한 문제 상황
AI 활용 기록은 단순히 “AI를 사용했다”라고 적는 것으로는 부족했다. 과제 조건은 문제 상황, 프롬프트, 채택/거부 이유, 배운 점을 요구하기 때문에 지금까지의 작업 기록을 보고서 문장으로 재구성해야 했다.

## 4. 채택한 내용
- AI 활용 보고서 목차를 과제 조건과 동일하게 구성한다.
- 단계별 프롬프트 원문을 근거 자료로 사용한다.
- 채택/거부 사례를 표로 정리한다.
- API 키 저장 금지와 사이트맵 우선 원칙을 핵심 사례로 강조한다.
- React/Tailwind 계획을 HTML/CSS/JS 구현으로 직접 조정한 내용을 사용자 판단 사례로 넣는다.
- 직접 해결한 문제와 배운 점을 별도 항목으로 작성한다.

## 5. 거부하거나 수정한 내용
- AI가 답변 전문을 그대로 보고서에 붙여 넣자고 하면 수정한다. 보고서에는 요약과 분석이 필요하다.
- AI가 “AI가 구현을 모두 해결했다”는 식으로 표현하면 수정한다. 실제로는 사용자가 기준을 정하고 AI 답변을 선별했다.
- AI가 채택한 내용만 쓰고 거부한 내용을 생략하면 수정한다. 과제 조건상 거부/수정 이유가 필수다.
- AI가 API 키나 민감 정보를 예시로 포함하면 제거한다.
- AI가 보고서 문장을 너무 기술 문서처럼 쓰면 수정한다. 수업 과제 보고서에 맞게 이해하기 쉬운 표현으로 바꾼다.
- AI가 React/Tailwind로만 구현했다고 단정하면 수정한다. 실제 구현은 HTML/CSS/JS 기반이며, React식 컴포넌트 분리 아이디어만 참고했다.

## 6. 내가 직접 해결한 부분과 배운 점
AI를 활용할 때 가장 중요한 것은 질문을 구체적으로 나누고, 답변을 그대로 받아들이지 않는 것이라는 점을 배웠다. 사이트맵 우선순위, API 키 보안, 과제 제출 범위, 보고서 캡처 기준은 직접 판단해야 했다. AI는 빠르게 대안을 제시해주지만, 최종적으로 어떤 제안을 채택하고 무엇을 제외할지는 프로젝트 목적과 과제 조건을 기준으로 결정해야 한다는 점을 알게 되었다. 특히 React/Tailwind 구조 제안은 참고했지만, 최종 제출 편의성과 단순 실행을 고려해 HTML/CSS/JS 기반으로 직접 수정했다.

## 9. 후속 수정 반영
10번 기록에서 최종 UI와 API 흐름이 다시 조정되었다. 보고서에는 “초기 계획은 두 API 키 모두 매번 입력하는 방식이었지만, 후속 요구사항에 따라 OpenWeatherMap은 과제용 키를 내장하고 OpenRouter 키만 매번 입력하도록 수정했다”고 적는다. 또한 “AI 모델은 사용자가 바꾸는 방식에서 GPT 5.4 nano 고정 방식으로 변경했고, 모바일 대응은 제외하고 데스크톱 제출 화면의 밀도를 높였다”고 정리한다. 이 변경은 AI가 임의로 한 것이 아니라, 사용자가 제출 방식과 시연 흐름을 고려해 직접 판단한 수정 사례로 기록한다.

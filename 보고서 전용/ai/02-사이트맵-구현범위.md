# 02 사이트맵 구현 범위

## 1. 사용자 프롬프트 원문
이전 단계에서 `사이트맵.pdf`를 최우선 기준으로 삼기로 했다. 이제 사이트맵에 그려진 LIFE CYCLE 웹앱을 실제 React 화면 구조로 바꿔줘.

반드시 지켜야 할 조건은 다음과 같다.
- 사이트맵에 보이는 구조를 먼저 구현 기준으로 삼는다.
- 메인 화면에는 LIFE CYCLE 로고, TODO, 습관, 일기, AI, 라이트/다크 모드 nav가 있어야 한다.
- 로고를 클릭하면 메인 화면으로 돌아오도록 한다.
- TODO 화면은 오늘 할 일, 기한, 우선순위, 폴더 구분이 드러나야 한다.
- 습관 화면은 오늘 해야 할 습관과 달성률이 보여야 한다.
- 일기/캘린더 화면은 월 이동 버튼, 날짜별 일정 점 표시, 날짜 클릭 후 일정과 한 줄 일기 표시가 있어야 한다.
- AI 화면은 일정 클릭 전과 일정 클릭 후의 브리핑 흐름을 구분해야 한다.
- 라이트/다크 모드 토글은 nav 오른쪽에 둔다.

출력은 다음 형식으로 해줘.
1. 화면 목록
2. 각 화면의 역할
3. React 라우트 또는 탭 구조
4. 주요 컴포넌트 이름
5. 구현 우선순위
6. 사이트맵에는 없어서 제외할 내용

새로운 기능을 임의로 추가하지 말고, 사이트맵과 구상 초안본 안에서만 판단해줘.

## 2. AI 답변 요약
- 메인 화면은 `/` 또는 `home` 상태로 두고, 전체 앱의 요약 대시보드 역할을 한다.
- nav 항목은 TODO, 습관, 일기, AI, 테마 토글로 구성한다.
- TODO 상세 화면은 할 일 목록, 폴더, 우선순위, 기한, 완료 상태를 관리한다.
- 습관 상세 화면은 오늘의 습관, 완료 체크, 달성률, 스크롤 가능한 목록을 제공한다.
- 일기 화면은 캘린더와 연결하고, 날짜 클릭 시 일정과 한 줄 일기를 입력/확인할 수 있게 한다.
- AI 화면은 선택된 일정이 없을 때 오늘 전체 조언을, 일정이 선택되었을 때 해당 일정 브리핑을 제공한다.
- 구현 우선순위는 메인/nav/캘린더, TODO/습관, 일기, AI API 연동 순서로 둔다.

## 3. AI 활용이 필요한 문제 상황
사이트맵은 여러 화면이 선으로 연결된 형태라 실제 React 앱에서 라우트로 나눌지, 탭 상태로 나눌지, 메인 안에 요약을 둘지 판단이 필요했다. 화면 단위가 정리되지 않으면 구현과 보고서 화면설계서가 모두 흔들릴 수 있었다.

## 4. 채택한 내용
- 메인은 대시보드형 첫 화면으로 구성한다.
- nav 클릭으로 TODO, 습관, 일기, AI 상세 화면에 접근하게 한다.
- 캘린더 날짜 클릭 시 2분할 상태를 만들어 해당 날짜 일정과 일기를 함께 보여준다.
- AI 브리핑은 “일정 선택 전”과 “일정 선택 후” 상태를 나누어 설계한다.

## 5. 거부하거나 수정한 내용
- AI가 여러 페이지를 과도하게 늘리면 수정한다. 과제 보고서에 설명 가능한 수준으로 메인, TODO, 습관, 일기, AI 중심으로 제한한다.
- AI가 사이트맵에 없는 지도, 로그인, 알림 서버, 공유 기능을 추가하면 제외한다.
- AI가 캘린더 날짜 클릭 기능을 단순 조회로만 처리하면 수정한다. 반드시 일정과 한 줄 일기가 함께 보이는 상태로 만든다.
- AI가 nav를 텍스트 버튼 위주로 만들면 수정한다. 기존 `icon/` PNG를 활용하는 방향으로 바꾼다.
- AI가 AI 브리핑을 단순 고정 문구로만 만들면 수정한다. 실제 API 연동 단계와 연결될 수 있게 요청 데이터 구조를 남긴다.

## 6. 내가 직접 해결한 부분과 배운 점
사이트맵을 실제 앱 구조로 바꿀 때는 화면의 이름보다 사용자의 행동 흐름을 먼저 봐야 한다는 점을 배웠다. 특히 “날짜 클릭 후 캘린더 2분할”은 사이트맵에 명확히 표시된 핵심 상호작용이므로 반드시 구현해야 한다고 직접 판단했다.

# 05 로컬 데이터 기능

## 1. 사용자 프롬프트 원문
메인 대시보드 UI를 만든 다음, TODO, 습관, 일정, 한 줄 일기를 실제로 입력하고 저장할 수 있게 만들고 싶다. 서버나 데이터베이스는 쓰지 않고 브라우저 localStorage를 사용해줘.

요구사항은 다음과 같다.
- TODO는 제목, 완료 여부, 폴더, 우선순위, 기한을 가진다.
- TODO 폴더는 공부, 건강, 일정, 알바, 기타 정도로 시작한다.
- TODO는 추가, 완료 체크, 삭제가 가능해야 한다.
- 습관은 이름, 목표 요일, 오늘 완료 여부, 달성률을 가진다.
- 습관은 오늘 해야 할 습관 목록에서 체크할 수 있어야 한다.
- 일정은 날짜, 제목, 시간, 장소를 가진다.
- 캘린더 날짜에는 해당 날짜 일정 개수만큼 점을 표시한다.
- 한 줄 일기는 날짜별로 저장하고, 날짜를 다시 클릭하면 기존 내용을 볼 수 있어야 한다.
- API 키는 절대 localStorage, sessionStorage, URL, 코드에 저장하지 않는다.
- 저장 데이터와 API 키 데이터를 타입과 함수 차원에서 명확히 분리한다.

출력은 구현 코드와 함께, 어떤 데이터가 저장되고 어떤 데이터가 저장되지 않는지도 설명해줘.

## 2. AI 답변 요약
- `useLocalStorage` 훅을 만들어 TODO, 습관, 일정, 일기만 저장한다.
- TODO 타입에는 id, title, done, folder, priority, dueDate를 둔다.
- Habit 타입에는 id, name, targetDays, completedDates를 둔다.
- ScheduleEvent 타입에는 id, date, title, time, location을 둔다.
- DiaryEntry 타입에는 date, text, mood 또는 emoji를 둔다.
- API 키는 컴포넌트 state에서 요청 중에만 사용하고 저장 함수에는 전달하지 않는다.
- localStorage key는 `life-cycle.todos`, `life-cycle.habits`, `life-cycle.events`, `life-cycle.diaries`처럼 구분한다.

## 3. AI 활용이 필요한 문제 상황
브라우저에 저장해야 하는 데이터와 저장하면 안 되는 API 키를 분명히 구분해야 했다. 기능별 데이터 타입을 미리 잡지 않으면 나중에 AI 브리핑이나 보고서 작성에서 설명이 어려워질 수 있었다.

## 4. 채택한 내용
- TODO, 습관, 일정, 일기는 localStorage에 저장한다.
- API 키는 어떤 저장소에도 저장하지 않는다.
- 날짜를 기준으로 캘린더, 일정, 일기를 연결한다.
- 데이터 타입과 localStorage key를 기능별로 분리한다.
- React 훅 대신 순수 JavaScript의 `load`, `save`, 렌더 함수로 같은 저장 흐름을 구현한다.

## 5. 거부하거나 수정한 내용
- AI가 API 키를 편의를 위해 저장하자고 하면 거부한다.
- AI가 모든 데이터를 하나의 큰 localStorage key에 넣으려 하면 수정한다. 기능별로 나누어 관리한다.
- AI가 삭제 기능 없이 추가만 구현하면 수정한다. TODO와 일정은 관리 기능이 필요하다.
- AI가 일기를 오늘 날짜에만 저장하도록 만들면 수정한다. 캘린더에서 선택한 날짜별로 저장해야 한다.
- AI가 습관 달성률을 단순 고정 숫자로 처리하면 수정한다. 완료 여부를 기준으로 계산되게 한다.
- AI가 `useLocalStorage` 훅만을 전제로 설명하면 수정한다. 실제 구현에서는 정적 HTML 기반이므로 순수 JavaScript localStorage 함수로 대체한다.

## 6. 내가 직접 해결한 부분과 배운 점
localStorage는 간단하지만 데이터 구조를 잘못 잡으면 기능이 복잡해진다는 점을 배웠다. 또한 API 키는 앱 데이터와 성격이 다르기 때문에 저장 대상에서 명확히 제외해야 한다고 판단했다. React 훅을 쓰지 않아도 저장 key와 렌더 함수를 분리하면 같은 구조를 HTML 기반으로 구현할 수 있다는 점을 직접 확인했다.

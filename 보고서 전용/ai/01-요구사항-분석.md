# 01 요구사항 분석

## 1. 사용자 프롬프트 원문
나는 프론트엔드 웹디자인 기말 대체 과제로 `LIFE CYCLE`이라는 일정/습관/TODO/일기/AI 브리핑 웹앱을 만들려고 한다.

현재 참고 자료는 다음 순서로 중요도를 둔다.
1. `보고서 전용/참고 파일/구상본/사이트맵.pdf`를 최우선 기준으로 사용한다.
2. `보고서 전용/참고 파일/구상본/구상 초안본.rtf`는 사이트맵을 보충하는 설명 자료로만 사용한다.
3. `보고서 전용/참고 파일/과제 조건/기말 프로젝트 조건.rtf`는 제출물과 AI 활용 문서화 조건을 확인하는 기준으로 사용한다.
4. `보고서 전용/참고 파일/보고서/Web FinalReport_2026.pptx`와 `Web Report Sample.pdf`는 보고서 목차와 표현 방식 참고용으로 사용한다.

위 자료를 바탕으로 다음 내용을 정리해줘.
- 실제로 구현해야 하는 핵심 기능과 우선순위
- 보고서에 반드시 들어가야 하는 내용
- AI 활용 보고서에 기록해야 하는 문제 상황, 프롬프트, 채택/거부 이유, 배운 점
- 사이트맵과 구상 초안본이 충돌할 때 어떤 기준으로 선택해야 하는지
- 이번 프로젝트에서 너무 과하거나 제출 범위 밖이라 제외해야 하는 기능

답변은 구현 코드를 작성하지 말고, 요구사항 분석 문서처럼 정리해줘. 이후 React 구현 단계에서 그대로 기준으로 사용할 수 있게 화면, 기능, 보고서 항목을 분리해서 작성해줘.

## 2. AI 답변 요약
- 최종 구현 기준은 사이트맵으로 두고, 구상 초안본은 기능 의도와 세부 설명을 보충하는 자료로 사용한다.
- 핵심 화면은 메인 대시보드, TODO/소규모 캘린더, 습관, 일기, AI 브리핑으로 구분한다.
- 메인 대시보드는 LIFE CYCLE 로고, 상단 nav, TODO 요약, 습관 요약, 캘린더, AI 브리핑이 한 화면에 보이는 구조로 해석한다.
- 과제 보고서에는 주제, 주요 내용, 기능 요약, 기획 일정, 참고 자료, 사이트맵, 화면설계서, 구현 결과, 수행 소감을 포함한다.
- AI 활용 보고서에는 AI를 사용한 이유, 프롬프트 원문, AI 답변 중 채택/거부한 내용과 이유, 직접 해결한 문제와 배운 점을 기록한다.
- API 키는 사용자 입력 방식으로 처리하고 저장하지 않는 방향이 적절하다.
- 범위 밖 기능은 구현보다 보고서에서 “추후 확장”으로 처리하는 것이 좋다.

## 3. AI 활용이 필요한 문제 상황
프로젝트 자료가 사이트맵, 구상 초안본, 과제 조건, 보고서 샘플로 나뉘어 있어 처음부터 구현에 들어가면 범위가 흔들릴 수 있었다. 특히 사이트맵은 손그림 형태라 기능 우선순위와 실제 화면 구조를 정리해줄 기준이 필요했다. 또한 과제 조건에서 AI 활용 문서화를 필수로 요구하므로, 구현 전부터 어떤 프롬프트와 답변을 남겨야 하는지 설계해야 했다.

## 4. 채택한 내용
- 사이트맵을 최우선 기준으로 삼는다.
- LIFE CYCLE의 핵심 기능을 TODO/소규모 캘린더, 습관, 일기, AI 브리핑으로 나눈다.
- 보고서 목차는 샘플 PPT 흐름을 따른다.
- AI 활용 기록은 작업 단계마다 별도 파일로 남긴다.
- API 키는 사용자가 기능을 실행할 때마다 입력하고 저장하지 않는 원칙을 둔다.

## 5. 거부하거나 수정한 내용
- AI가 구상 초안본에 있는 모든 아이디어를 무조건 구현하려고 하면 제외한다. 사이트맵에 없는 기능은 우선순위를 낮추거나 보고서의 추후 개선 항목으로 돌린다.
- AI가 서버, 데이터베이스, 로그인, 회원가입처럼 과제 범위를 크게 늘리는 제안을 하면 제외한다.
- AI가 API 키를 `.env`, localStorage, sessionStorage, 코드 상수에 저장하자고 하면 모두 거부하고 “매번 직접 입력” 방식으로 수정한다.
- AI가 보고서 조건을 생략하고 구현 기능만 정리하면 수정한다. 과제 조건의 AI 활용 문서화 항목을 반드시 포함시킨다.
- AI가 사이트맵보다 예시 프로젝트나 일반적인 생산성 앱 패턴을 우선하면 수정한다. 판단 기준은 항상 사이트맵이 먼저다.

## 6. 내가 직접 해결한 부분과 배운 점
자료를 단순히 읽는 것보다 우선순위를 정하는 과정이 중요하다는 점을 배웠다. 특히 사이트맵은 실제 화면 배치와 사용자 흐름을 보여주는 자료이므로 구현 기준으로 삼고, 구상 초안본은 기능의 의도를 이해하는 보조 자료로 사용하는 것이 적절하다고 판단했다. 또한 AI 활용 보고서는 마지막에 몰아서 쓰면 누락될 수 있으므로, 작업 단계마다 기록을 남기기로 결정했다.

## 7. 다음 단계
사이트맵을 실제 React 라우트, 컴포넌트, 화면 상태로 변환한다. 이때 사이트맵에 보이는 메인 화면과 nav 구조를 먼저 확정한다.

# 07 날씨 AI 브리핑 연동

## 1. 사용자 프롬프트 원문
이전 단계에서 API 키를 매번 직접 입력하고 저장하지 않는 UI를 설계했다. 이제 그 조건을 지키면서 OpenWeatherMap과 OpenRouter를 실제 AI 브리핑 기능에 연결해줘.

요구사항은 다음과 같다.
- 사용자가 입력한 OpenWeatherMap API 키로 지역명을 좌표로 변환한다.
- 지역명 변환에는 OpenWeatherMap Geocoding API를 사용한다.
- 변환된 좌표로 Current Weather API를 호출해서 현재 날씨를 가져온다.
- 날씨 정보는 온도, 체감온도, 날씨 설명, 습도, 바람 정보를 사용한다.
- 사용자가 입력한 OpenRouter API 키로 Chat Completion API를 호출한다.
- OpenRouter 프롬프트에는 선택된 일정, 장소, 시간, 오늘 TODO 요약, 습관 달성 상태, 날씨 정보를 포함한다.
- 일정이 선택되지 않았으면 오늘 전체 일정과 하루 조언을 요청한다.
- 일정이 선택되었으면 해당 일정에 대한 준비물, 시간 관리, 날씨 기반 조언을 요청한다.
- 요청 성공 후 API 키 state를 비운다.
- 요청 실패 후에도 API 키 state를 비운다.
- API 키 오류, 지역명 오류, 날씨 API 오류, OpenRouter 오류, 모델명 오류를 구분해서 사용자에게 보여준다.
- API 키는 절대 저장하거나 로그로 출력하지 않는다.

코드를 작성할 때 `fetchWeatherByLocation`, `createAIBriefing`처럼 역할이 드러나는 함수로 나눠줘. 보고서에 설명할 수 있도록 요청 흐름도 함께 정리해줘.

## 2. AI 답변 요약
- OpenWeatherMap Geocoding API로 지역명을 좌표로 변환한 뒤 Current Weather API를 호출한다.
- 날씨 응답에서 온도, 체감온도, 습도, 바람, 설명을 정리해 `WeatherSummary`로 만든다.
- OpenRouter 요청에는 일정과 날씨, TODO, 습관 정보를 함께 넣어 개인화된 브리핑을 만든다.
- 일정 선택 전에는 하루 전체 조언을, 일정 선택 후에는 특정 일정 중심 조언을 생성한다.
- API 호출 함수는 역할별로 분리하고 오류 종류를 명확히 반환한다.
- `finally` 또는 요청 종료 처리에서 API 키 state를 초기화한다.
- 실패 상태도 UI에 표시하여 사용자가 원인을 알 수 있게 한다.

## 3. AI 활용이 필요한 문제 상황
AI 브리핑 기능은 두 개의 외부 API를 순서대로 사용한다. 날씨 정보를 먼저 가져온 뒤 그 결과를 OpenRouter 프롬프트에 넣어야 하므로 요청 흐름이 복잡했다. 또한 키 저장 금지 조건을 지키면서 오류 처리까지 해야 했다.

## 4. 채택한 내용
- 지역명 → 좌표 → 날씨 → AI 브리핑 순서로 처리한다.
- OpenWeatherMap 호출과 OpenRouter 호출을 별도 함수로 분리한다.
- 일정 선택 여부에 따라 AI 프롬프트를 다르게 만든다.
- 오류 상태를 구분하여 UI에 표시한다.
- 요청 종료 후 API 키를 비운다.

## 5. 거부하거나 수정한 내용
- AI가 날씨 API 없이 고정 더미 날씨만 사용하려 하면 수정한다. 이 단계는 실제 API 연동 단계다.
- AI가 OpenRouter 프롬프트에 API 키나 민감 정보를 포함하면 즉시 제거한다.
- AI가 모든 오류를 “실패했습니다” 하나로 처리하면 수정한다. 보고서에서 설명할 수 있게 오류 종류를 구분해야 한다.
- AI가 키 초기화를 성공 시에만 하게 만들면 수정한다. 실패 시에도 반드시 초기화한다.
- AI가 모델명을 코드에 고정하고 사용자가 바꿀 수 없게 만들면 수정한다. 사용자가 입력한 모델명을 사용할 수 있게 한다.
- AI가 CORS 문제가 생긴다고 바로 백엔드 서버를 추가하려 하면 우선 보류한다. 과제 범위를 넘기므로 브라우저 직접 호출을 먼저 시도한다.

## 6. 내가 직접 해결한 부분과 배운 점
실제 API 연동은 단순히 fetch 한 번으로 끝나는 것이 아니라 데이터 흐름과 실패 흐름을 함께 설계해야 한다는 점을 배웠다. 특히 날씨 API 결과가 AI 브리핑의 입력이 되기 때문에 순서와 데이터 변환이 중요하다는 점을 직접 확인했다.

## 9. 후속 수정 반영
초기에는 OpenRouter 모델명을 사용자가 입력할 수 있게 두는 방향이었지만, 후속 요구사항에서 GPT 5.4 nano 하나로 한정하기로 변경했다. 따라서 최종 구현에서는 모델 입력칸을 제거하고 `openai/gpt-5.4-nano` 고정값만 사용한다. OpenWeatherMap 키도 사용자가 매번 입력하는 방식에서 과제용 내장 키 방식으로 바뀌었고, 캘린더 2분할 일정 버튼을 누르면 해당 일정이 바로 브리핑 대상으로 선택되도록 수정했다.

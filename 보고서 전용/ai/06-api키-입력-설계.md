# 06 API 키 입력 설계

## 1. 사용자 프롬프트 원문
이제 AI 브리핑 기능에서 OpenWeatherMap과 OpenRouter API를 사용할 준비를 하려고 한다. 하지만 이 프로젝트에서는 API 키를 파일이나 브라우저 저장소에 저장하지 않고, 사용자가 AI 브리핑을 실행할 때마다 직접 입력하도록 만들 계획이다.

다음 조건을 만족하는 UI와 상태 흐름을 설계해줘.
- AI 브리핑 실행 버튼을 누르면 API 키 입력 모달을 띄운다.
- 모달에는 OpenWeatherMap API Key, OpenRouter API Key, 지역명, OpenRouter 모델명, 선택 일정 정보가 들어간다.
- API 키 입력 필드는 `password` 타입으로 만든다.
- 사용자가 실행 버튼을 누르면 이번 요청에만 키를 사용한다.
- 요청이 끝나면 성공/실패와 관계없이 API 키 state를 빈 문자열로 초기화한다.
- API 키는 localStorage, sessionStorage, URL query, `.env`, 코드 상수, 보고서 캡처에 남기지 않는다.
- 저장 체크박스나 “다음에도 사용” 기능은 만들지 않는다.
- API 키가 비어 있으면 실제 요청을 보내지 않고 입력 필요 메시지를 보여준다.
- 오류가 발생해도 키 값이 화면에 노출되지 않게 한다.

출력은 컴포넌트 설계, 상태 흐름, 보안상 제외할 기능, 보고서에 적을 설명 문장으로 나누어 작성해줘.

## 2. AI 답변 요약
- `ApiKeyDialog` 컴포넌트를 만들어 AI 브리핑 실행 시 표시한다.
- 입력 필드는 OpenWeatherMap 키, OpenRouter 키, 지역명, 모델명으로 구성한다.
- API 키는 `useState`로만 관리하고 요청 직후 초기화한다.
- “이번 요청에만 사용되며 저장되지 않습니다”라는 안내 문구를 표시한다.
- 키가 비어 있으면 요청을 막고 사용자에게 입력 필요 상태를 보여준다.
- 저장 옵션, 자동 완성 안내, 브라우저 저장 기능은 제공하지 않는다.
- 보고서에는 API 키 보안을 고려해 매번 입력 방식을 채택했다고 적는다.

## 3. AI 활용이 필요한 문제 상황
실제 API 연동을 하려면 키가 필요하지만, 프론트엔드 과제에서는 키가 코드에 남거나 제출 파일에 포함되면 문제가 될 수 있었다. 따라서 사용자가 매번 직접 입력하면서도 앱 사용 흐름이 어색하지 않도록 설계해야 했다.

## 4. 채택한 내용
- API 키 입력 모달을 만든다.
- 키는 `password` 타입으로 입력받는다.
- 요청이 끝나면 키 state를 초기화한다.
- 키를 저장하지 않는다는 안내 문구를 보여준다.
- API 키 누락 상태를 별도 오류로 처리한다.

## 5. 거부하거나 수정한 내용
- AI가 `.env`에 API 키를 넣는 방식을 제안하면 이번 과제 조건에 맞지 않으므로 수정한다.
- AI가 localStorage나 sessionStorage에 API 키를 저장하자고 하면 거부한다.
- AI가 “편의를 위해 키 기억하기” 체크박스를 만들면 제거한다.
- AI가 API 키를 콘솔 로그로 출력하거나 오류 메시지에 포함하면 수정한다.
- AI가 브리핑 요청 실패 후 입력 키를 계속 보관하게 만들면 수정한다. 실패해도 키는 비워야 한다.
- AI가 보고서 캡처에 실제 키가 보이도록 예시를 만들면 수정한다. 보고서에는 마스킹된 상태만 사용한다.

## 6. 내가 직접 해결한 부분과 배운 점
API를 사용하는 기능은 편리함만 생각하면 키를 저장하고 싶어질 수 있지만, 제출 프로젝트에서는 보안과 개인정보가 더 중요하다는 점을 배웠다. 따라서 사용자가 번거롭더라도 매번 직접 입력하는 방식을 선택했고, 이 판단을 AI 활용 보고서에서 중요한 채택/수정 사례로 쓰기로 했다.

## 7. 다음 단계
사용자가 입력한 API 키를 이번 요청에만 사용하여 OpenWeatherMap 날씨 조회와 OpenRouter AI 브리핑을 실제로 연결한다.

## 8. 실제 진행 시나리오
이 단계에서는 API 호출 자체보다 키 입력 흐름을 먼저 완성한다. 다음 단계에서 AI에게 “06 API 키 입력 설계의 저장 금지 조건을 절대 깨지 말라”고 명시하여 실제 연동 코드에서도 키가 남지 않게 한다.

## 9. 후속 수정 반영
후속 요구사항에서 OpenWeatherMap API 키는 사용자가 직접 제공한 과제용 키를 코드에 미리 넣는 방식으로 변경되었다. 따라서 최종 구현에서는 OpenWeatherMap 키 입력칸을 제거하고, OpenRouter API 키만 실행 시 입력하도록 수정했다. 기존 보안 원칙은 OpenRouter 키에 계속 적용하며, OpenWeatherMap 키 원문은 보고서용 기록에는 마스킹한다.

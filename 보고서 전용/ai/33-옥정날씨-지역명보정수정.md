# 옥정 날씨 지역명 보정 수정

## 1. 사용자 프롬프트 원문
옥정 날씨 왜 확인 안되는지 이유만 먼저 알려줘.

너가 제안한 해결방안 적용해서 수정해줘.

## 2. AI 답변 요약
- 기존 날씨 조회는 사용자가 입력한 지역명을 OpenWeatherMap 지오코딩 API에 그대로 전달하고 있었다.
- `옥정`처럼 짧은 동네명은 OpenWeatherMap에서 위치를 찾지 못할 가능성이 높다고 판단했다.
- API 키 문제보다는 지역명 해석 문제로 보고, `옥정`을 `양주시 옥정동` 기준으로 보정하는 방식이 적절하다고 설명했다.
- 수정 시에는 지오코딩 검색어 후보를 늘리고, 그래도 실패하면 좌표 기반 fallback을 사용하는 방식을 제안했다.
- 브리핑 기능 자체는 유지하고 날씨 조회 함수 내부만 보완하는 방향으로 구현했다.

## 3. AI 활용이 필요한 문제 상황
일정 지역에 `옥정`을 넣었을 때 날씨 확인이 되지 않아, API 키 문제인지 지역명 처리 문제인지 원인을 구분해야 했다. 또한 과제 웹사이트에서는 사용자가 상세 행정구역을 매번 정확히 입력하지 않아도 브리핑이 자연스럽게 작동해야 했기 때문에 지역명 보정 로직이 필요했다.

## 4. 채택한 내용
- `옥정`, `옥정동` 입력 시 `옥정동, 양주시, KR`, `양주시 옥정동, KR`을 추가 검색어로 사용하도록 했다.
- OpenWeatherMap 지오코딩 결과가 없을 경우 `양주시 옥정동` 좌표를 fallback으로 사용하도록 했다.
- 기존처럼 지역이 없으면 날씨 없이 일정 중심 브리핑을 생성하는 흐름은 유지했다.
- OpenWeatherMap API 키 저장 방식이나 OpenRouter 키 입력 방식은 변경하지 않았다.

## 5. 거부하거나 수정한 내용
- 모든 지역명을 자동 추론하는 큰 기능은 과제 범위를 벗어나므로 넣지 않았다.
- 사용자가 직접 위도/경도를 입력하는 방식은 UI가 복잡해지고 사이트맵 흐름과 맞지 않아 제외했다.
- 지역 입력 UI 전체를 변경하지 않고, 현재 일정 추가에서 입력한 지역명을 내부적으로 보정하는 방식으로 수정했다.

## 6. 내가 직접 해결한 부분과 배운 점
옥정 날씨 실패 원인이 API 키가 아니라 외부 API의 지역명 검색 한계라는 점을 확인했다. AI 제안을 그대로 크게 확장하지 않고, 과제에서 실제로 필요한 `옥정` 케이스만 작게 보완하는 방식으로 적용했다. 외부 API를 사용할 때는 사용자가 입력한 값을 그대로 보내는 것보다, 서비스가 이해할 수 있는 형태로 보정하거나 실패 시 대체 경로를 준비해야 한다는 점을 배웠다.

## 7. 다음 단계
브라우저에서 `옥정` 또는 `옥정동` 지역이 포함된 일정을 선택한 뒤 AI 브리핑을 실행해 날씨 정보가 포함되는지 확인한다. 네트워크/API 상태 문제로 실패하더라도 브리핑 자체는 중단되지 않는지 함께 점검한다.
